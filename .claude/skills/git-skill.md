# Git Skill Guide

## 적용 조건 (자동 활성화)
- 키워드: 커밋, 브랜치, 푸시, 배포, GitHub, merge, pull, tag
- 파일 경로: .git/

---

## 1. 브랜치 전략
- main: 항상 배포 가능한 안정 버전 (GitHub Pages 연결)
- develop: 개발 통합 브랜치
- feature/xxx: 기능 단위 개발 브랜치
- hotfix/xxx: 긴급 버그 수정 브랜치

## 2. 브랜치 규칙
- main 브랜치에 직접 커밋 절대 금지
- 모든 기능 개발은 feature/ 브랜치에서 시작
- 기능 완료 후 develop → main 순서로 병합
- 병합 전 반드시 사용자 확인

## 3. 커밋 메시지 규칙 (Conventional Commits)
- feat: 새 기능 추가
- fix: 버그 수정
- design: UI/디자인 변경
- docs: 문서 수정
- refactor: 코드 구조 개선
- chore: 기타 작업

## 4. 버전 태그 규칙 (Semantic Versioning)
- v1.0.0 → Major: 대규모 변경
- v1.1.0 → Minor: 기능 추가
- v1.0.1 → Patch: 버그 수정

### 버전 계획
- v1.0.0: MVP (카드 도감 기본) ✅ 완료
- 이후 버전: TBD (사용자와 협의 후 결정)

## 5. 추가 예정 기능 목록 (우선순위 TBD)
- 카드 도감 고도화 (실제 API 데이터, 상세 필터, 한/영 전환, 팩 오픈율)
- 덱 빌더
- 카드팩 오픈 시뮬레이션
- 커뮤니티
- 뉴스
- 기타 (사용자 추가 예정)

## 6. 커밋 타이밍 규칙
- 기능 하나 완료 시 커밋
- 하루 작업 마칠 때 커밋
- 큰 변경 전 백업용 커밋
- 작업 중간에 임의로 커밋 금지

## 7. 배포 규칙
- GitHub Pages: main 브랜치 자동 배포
- 배포 전 반드시 develop에서 테스트 확인
- 배포 후 실제 URL에서 정상 동작 확인

## 8. 작업 완료 후 체크리스트
- [ ] 커밋 메시지가 규칙을 따르는가?
- [ ] main 브랜치에 직접 커밋하지 않았는가?
- [ ] 버전 태그가 올바르게 생성되었는가?
- [ ] 사용자 확인 후 푸시했는가?
- [ ] 배포 후 실제 URL에서 확인했는가?
